## 1. 产品愿景与定位

### 1.1 产品愿景

**Lamp** 是一款面向中文作家的**AI桌面富文本编辑器**，核心理念是"无干扰 + AI辅助"。它不是泛化的AI写作助手，而是一款专注于**长篇创作（小说/剧本/随笔）**的沉浸式工具——兼具现代编辑器的专业体验与AI的创作加速能力。

### 1.2 当前定位

Lamp 目前的定位是一个**To C 商业级产品**：
- ✅ 富文本编辑（TipTap）
- ✅ 插件系统（架构已就绪）
- ✅ 基础AI辅助（4个操作）
- ✅ 工作区文件树
- ✅ 自动保存 / 窗口管理

**缺失的核心能力**：创作流程管理、AI深度集成、输出发布。

### 1.3 目标定位（v2.0）

| 维度 | 当前 | 目标 |
|------|------|------|
| 用户类型 | 尝鲜用户 | 严肃创作者 |
| 文档规模 | 单篇文档 | 多文档项目 |
| AI使用方式 | 被动辅助 | 主动协作 |
| 输出能力 | 手动另存 | 一键多格式发布 |
| 目标市场 | 通用 | AI小说/剧本垂直领域 |

## 3. 现状问题审计

> 以下问题按 **严重程度** 排列。P0 为阻塞性缺陷，P1 为核心体验缺失，P2 为重要功能空白，P3 为增强性需求。

### P0 — 阻塞性缺陷

#### P0-1. 无项目管理能力（文档孤立）

**问题**：当前以单文档为操作单位，无"项目"概念。
- 写小说时 dozens of 章节散布在工作区，缺乏逻辑分组
- 无法将多个文档标记为同一作品的组成部分
- 切换章节/场景需要手动切换标签页或切换文件

**影响**：无法支撑任何超过3章的严肃创作。

**参考竞品**：Sudowrite（Project）、Plottr（Book）、Manuskript（Manuscript）均以项目为单位组织。

---

#### P0-2. 无大纲/结构视图

**问题**：编辑器只有线性文本流，没有上层结构视角。

**影响**：创作者无法俯瞰全作结构，无法快速跳转到特定章节/场景，写作过程中极易"迷路"。

**参考竞品**：Manuskript 的 Index Card 视图、Plottr 的节拍表、Sudowrite 的 Story Engine。

---

#### P0-3. 专注模式形同虚设

**问题**：现有"专注模式"仅等价于全屏（F11），没有任何 UI 隐藏或段落高亮。

**影响**：干扰源（菜单、状态栏、文件树）仍然可见，无法实现真正的沉浸式写作。

**参考竞品**：Focus Writer 的 Typewriter Mode（光标居中，隐藏所有 UI）、iA Writer 的透明模式。

---

### P1 — 核心体验缺失

#### P1-1. 无写作目标与统计

**问题**：没有任何字数统计面板、写作目标设置或会话统计。

**影响**：长篇创作中无法跟踪进度，写作动力不足。这是严肃创作者的刚性需求。

**参考竞品**：Focus Writer（每日/项目字数目标）、Manuskript（会话统计）、NovelAI（实时字数）。

---

#### P1-2. AI能力单薄（仅有4个操作）

**问题**：AI功能仅覆盖"润色/扩写/续写/概述"四个场景。

**缺失的关键AI能力**：
- 基于项目上下文的记忆（角色设定、世界观规则——NovelAI Lorebook 类）
- 多轮对话式AI协作（头脑风暴、情节推理、角色对话生成）
- AI生成大纲/节拍表
- AI检测情节漏洞/重复
- AI风格模仿（学习已有文风续写）
- AI生成角色对白
- 自定义AI提示词模板

**参考竞品**：Sudowrite（Plot Threads、Feedback）、NovelAI（Lorebook、Memory）、Squibler（AI Outline）。

---

#### P1-3. 查找替换功能缺失

**问题**：完全没有 Find（Ctrl+F）和 Replace 功能。

**影响**：在长篇文档中无法定位关键词，无法批量替换用语。这是所有现代编辑器的基本能力。

---

#### P1-4. 快捷键文档化程度低

**问题**：部分快捷键已实现（Ctrl+S/O/Z/Y/F11/Ctrl+Shift+P），但：
- 没有快捷键设置界面（Settings > Shortcuts 是空壳）
- 没有快捷键提示悬浮
- 很多应有快捷键缺失（New File、Find/Replace 等）

**参考竞品**：所有成熟编辑器均内置快捷键自定义。

---

#### P1-5. 导出能力为零

**问题**：只能另存为 `.lmph`/`.html`/`.md`/`.txt`，没有多格式导出（ePub、PDF）。

**影响**：无法作为实际创作-发布工作流的一环。创作者完成编辑后必须借助 Calibre 等外部工具转换。

**参考竞品**：Sudowrite（ePub/PDF）、Squibler（ePub/PDF/mobi）、Manuskript（odt/pdf）。

---

### P2 — 重要功能空白

#### P2-1. 无人物/世界观数据库

**问题**：无法在编辑器内管理角色卡、地点、势力等创作素材。

**参考竞品**：Campfire Writing（角色卡+关系图）、Squibler（Character Builder）、World Anvil（Wiki式世界观）。

---

#### P2-2. 自动保存无冲突处理

**问题**：自动保存至 `.lampsave`，但当外部修改了同一文件时无冲突提示/合并机制。

**参考竞品**：Notion、Obsidian 的冲突解决界面。

---

#### P2-3. 无暗色主题

**问题**：整个应用只有浅色主题。

**影响**：长时间写作的用户体验差，尤其夜间创作者。这是几乎所有竞品的标配。

**参考竞品**：Sudowrite、NovelAI、Plottr 均支持深色模式。

---

#### P2-4. 无分割视图/并排编辑

**问题**：无法同时查看同一文档的两个位置，或同时打开两个文件并排比较。

**参考竞品**：VS Code、Typora 的分割视图。

---

#### P2-5. AI模型配置过于简陋

**问题**：AI设置仅支持 Base URL + Model + API Key，无：
- Temperature/top_p 参数控制
- 每次请求的 Token 限制
- 多模型切换
- 请求超时配置

**参考竞品**：NovelAI 的详细参数面板。

---

#### P2-6. 无最近文件列表

**问题**：没有"最近打开"文件/项目的快速入口，每次都要通过原生对话框导航。

**参考竞品**：VS Code、Notion 的 Recent Files 面板。

---

### P3 — 增强性需求

#### P3-1. 无拼写/语法检查

**问题**：无任何拼写检查、语法提示或同义词替换功能。

**参考竞品**：LanguageTool 集成（Sudowrite）、Grammarly 集成。

---

#### P3-2. 无片段/模板系统

**问题**：无法存储和复用常用文本片段（开场白、地点描写模板等）。

**参考竞品**：VS Code Snippets、Typora 模板。

---

#### P3-3. 无只读/评论协作

**问题**：无法将文档分享为只读链接，无法添加评论/批注。

**参考竞品**：Google Docs 风格的批注系统（Sudowrite Feedback 功能）。

---

#### P3-4. 状态栏信息不完整

**问题**：状态栏仅显示：工作区标识、光标位置、UTF-8。

**缺失**：字数（含选中/全文）、章节标题、写作进度条、API连接状态。

---

#### P3-5. 插件系统对用户不可见

**问题**：插件系统架构已完整实现，但：
- 没有插件商店 UI
- 没有插件启用/禁用管理界面（Settings > Plugins 是空壳）
- 没有内置插件的自定义配置（如 AI 提示词模板管理）

---

#### P3-6. 无撤销历史/快照

**问题**：TipTap 的 Undo/Redo 只作用于当前会话，无法浏览历史版本。

**参考竞品**：Notion 的 History、Manuskript 的快照功能。

## 6. 技术建议

### 6.1 架构层面

1. **项目元数据存储**：使用 `{.project}/.lamp/metadata.json`，遵循[SPEC.md](./SPEC.md)约定
2. **Lorebook 注入策略**：AI 请求前按相关性检索 top-K 实体，拼接到 System Prompt（需实现简单的向量相似度或 BM25 检索）
3. **ePub 导出**：可引入 `jszip` + ` epub-gen`（npm 包）实现，无需 Rust 端
4. **拼写检查**：考虑 `https://languagetool.org/` HTTP API 或 WebAssembly 的 `spellcheck` crate`
5. **版本快照**：使用 `git` 或简化的 JSON diff 存储（Rust 端实现）

### 6.2 TipTap 扩展建议

| 扩展 | 用途 | 优先级 |
|------|------|--------|
| `CharacterCount` | 字数统计 | P1 |
| `Search` | 查找高亮（配合 SEA-01） | P1 |
| `Placeholder` | Typewriter Mode 占位符 | P0 |
| 自定义 `FocusMode` 节点 | 段落高亮 | P1 |
| `TableOfContents` | 大纲生成 | P0 |

### 6.3 性能注意事项

- Lorebook 向量检索在 Rust 端实现（使用 `tantivy` 或 `meilisearch` Rust SDK），避免前端瓶颈
- 文件树使用虚拟滚动（`el-tree-v2` 已在用），确保万级文件不卡顿
- AI 请求加超时控制（建议 30s），避免 UI 长时间阻塞

## 7. 附录

### A. 竞品功能详细参考

**NovelAI 关键特性（重点参考）**：
- Lorebook：key-value 百科，可按关键词触发注入上下文
- Memory：全局写作记忆，在每次生成时携带
- Style Presets：预设写作风格（Second Person、Assistants等）
- Image Generation：AI 生成书籍封面（次要）

**Sudowrite 关键特性（重点参考）**：
- Plot Threads：情节线管理（主线/支线/伏笔），可标注在文本中
- Describe：生成感官描写（视觉/听觉/嗅觉等）
- Feedback：AI 扮演Beta读者，给出反馈意见
- First Draft：基于大纲生成完整章节草稿

**Manuskript 关键特性（开源参考）**：
- Snowflake Mode：从一句话 → 一段话 → 一页纸 → 章节的雪球式大纲法
- Index Card：每个场景一张卡，可拖拽排序
- Summary View：全局摘要视图，每张卡片一行总结
